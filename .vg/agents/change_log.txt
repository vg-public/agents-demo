CHANGE 1 — NEW AGENT: sql-analysis.agent.md
================================================================================
updates

File      : .github/agents/sql-analysis.agent.md
Operation : CREATED (net-new file)
Replaces  : @sql-data was previously the only SQL-related agent; it generated DDL
            and seed data but was NOT wired into the story workflow. @sql-analysis
            does not replace @sql-data — they coexist. @sql-data still generates
            schemas; @sql-analysis inspects SQL in story content before coding begins.

WHY THIS AGENT WAS ADDED
─────────────────────────
When a developer writes a user story, they often embed reference SQL queries to
describe the data access pattern (e.g. SELECT with WHERE clauses, JOINs, filters).
Previously, that SQL was ignored by the orchestrator. @java-api-dev had to infer
the mapping on its own, leading to mismatches between the DDL schema and the entity
fields generated. This agent bridges that gap.

HOW IT WORKS
────────────
1. FAST-PASS RULE (most important): The agent FIRST scans the story content for
   SQL signals (SELECT/INSERT/UPDATE/DELETE keywords, UPPER_SNAKE_CASE identifiers,
   column references, or words like "query", "table", "column").
   - If NO signals found → outputs the fast-pass message and stops immediately.
     This keeps the workflow fast for stories with no SQL content.
   - If ANY signal found → proceeds to full analysis.

2. FULL ANALYSIS (only when SQL is found):
   - Reads src/main/resources/db/oracle-ddl.sql (the authoritative DDL reference).
   - Cross-references story SQL against actual DDL definitions.
   - Produces a structured mapping plan covering:
       * Table → Entity class name
       * Column → Java field name + Java type + JPA annotations
       * Foreign key constraints → JPA relationship annotations (@ManyToOne, etc.)
       * Sequence definitions → @SequenceGenerator allocationSize
       * WHERE/JOIN/ORDER BY patterns → candidate repository method signatures

3. OUTPUT is passed verbatim as context to @java-api-dev in the next workflow step.
   @java-api-dev treats the SQL Analysis Plan as the authoritative schema reference
   and uses it to generate correct entity fields and repository methods.

FRONTMATTER (copy exactly into target project)
───────────────────────────────────────────────
  tools: [read, search]
  (NO terminal access — read-only agent)
  argument-hint: "Pass the full story content. If a non-default DDL file path is
                  needed, include it — otherwise defaults to src/main/resources/db/oracle-ddl.sql."

ADAPTING TO ANOTHER PROJECT
────────────────────────────
- Change the default DDL path from src/main/resources/db/oracle-ddl.sql to your
  project's DDL reference file.
- If your project uses a different naming convention for sequences (e.g. SEQ_TABLE
  instead of TABLE_SEQ), update the JPA Annotations Summary section output format.
- The fast-pass message string can remain unchanged — it is parsed by the
  orchestrator to detect the fast-pass condition.

CHANGE 2 — NEW AGENT: maven-ops.agent.md
================================================================================

File      : .github/agents/maven-ops.agent.md
Operation : CREATED (net-new file)
Position  : Wired AFTER @testing and BEFORE @code-review in Story and Bug Fix workflows.

WHY THIS AGENT WAS ADDED
─────────────────────────
Previously, after @testing wrote tests, the orchestrator immediately invoked @code-review.
This meant that if the build was broken (compilation errors introduced by generated code,
or newly written tests that fail), the code review would run on a broken state. The
developer had no automated validation that the build was actually green.

@maven-ops fills this gap: it is a mandatory BUILD GATE step. Code review and git
operations only proceed after @maven-ops confirms compile + test success.

HOW IT WORKS
────────────
Sequential chain with hard stops. Each step only runs if the previous succeeds:

  mvn compile
      ↓ STOP on failure (reports all [ERROR] lines with file/line)
  mvn test
      ↓ STOP on failure (reports failing test class + message)
  [ASK USER] → Start server?
      ↓ Only on explicit yes
  mvn spring-boot:run  OR  Kill-and-restart PowerShell script

The kill-and-restart script:
  Get-NetTCPConnection -LocalPort 8080 → get PID → Stop-Process -Force →
  Start-Sleep 2 → mvn spring-boot:run

SERVER START IS ALWAYS GATED: The agent never starts or restarts the server
without the user explicitly typing "yes". This prevents accidentally killing a
running server during a routine build gate check.

CAPABILITIES
────────────
  compile          — mvn compile only
  test             — mvn compile then mvn test
  run              — compile + test + ask + spring-boot:run
  restart          — kill port 8080 + spring-boot:run (asks before executing)
  all              — compile → test → ask before run
  coverage         — mvn test jacoco:report (parses coverage summary)

FRONTMATTER
───────────
  tools: [terminal]
  (terminal-only agent — no file editing)

ADAPTING TO ANOTHER PROJECT
────────────────────────────
- Replace "AgentsDemoApplication" (in the "Started X" log watch) with your
  Spring Boot main class name.
- Port 8080 is hardcoded in the kill script; change if your app uses a different port.
- If not on Windows/PowerShell, the kill script needs updating to use lsof/kill.
- Add profile flags (e.g. -Dspring.profiles.active=dev) to the run command if needed.

CHANGE 3 — NEW AGENT: git-ops.agent.md
================================================================================

File      : .github/agents/git-ops.agent.md
Operation : CREATED (net-new file)
Position  : FINAL step in Story Implementation and Bug Fix workflows.

WHY THIS AGENT WAS ADDED
─────────────────────────
Previously there was no automated way to go from "tests pass" to "branch + PR on
GitHub". The developer had to manually run git commands after every story. This
agent automates the entire publish workflow while enforcing safety controls that
prevent accidental pushes to protected branches or commits with secrets.

HUMAN-IN-THE-LOOP DESIGN
─────────────────────────
This is the most safety-critical agent. It uses a MANDATORY CONSENT GATE: before
executing ANY git command, it displays a full summary (branch name, commit message,
files changed, PR title, target branch) and explicitly asks the human to type YES.

On NO or silence → outputs "GIT OPS: Aborted." and exits. No git commands run.
This design means git operations can never be triggered accidentally by an
orchestrator subagent invocation — a human must explicitly confirm every time.

PRE-FLIGHT CHECKS (run before showing consent gate)
────────────────────────────────────────────────────
1. mvn test -q           — verify tests still pass
2. git diff --check      — detect merge conflict markers
3. Scan staged diff for hardcoded credential patterns (passwords, API keys, secrets)
4. git status --short    — confirm no target/ artifacts are staged

If any check fails → report and STOP (consent gate is never shown).

BRANCH NAMING CONVENTION
────────────────────────
  Stories/Features : feature/<ticket>-<slug>   e.g. feature/EPIC-001-S004-product-search
  Bug Fixes        : bugfix/<ticket>-<slug>     e.g. bugfix/BUG-042-cart-rounding
  Hotfixes         : hotfix/<slug>              e.g. hotfix/fix-payment-timeout

Always branched from develop:
  git checkout develop → git pull origin develop → git checkout -b feature/...

COMMIT FORMAT (Conventional Commits)
─────────────────────────────────────
  feat(<scope>): <short description>

  <body: what changed and why>

  Fixes <ticket-id>

SAFETY GUARDS
─────────────
  - NEVER push to main or develop directly
  - NEVER git push --force
  - NEVER stage target/, *.class, generated sources
  - NEVER commit conflict markers
  - NEVER auto-retry a failed push

PR CREATION
───────────
  Uses GitHub CLI: gh pr create --base develop --title "..." --body "..."
  PR body template: ## What / ## Why / ## How / ## Testing

FRONTMATTER
───────────
  tools: [terminal, read, search]

ADAPTING TO ANOTHER PROJECT
────────────────────────────
- Change the base branch from "develop" to your default integration branch
  (e.g. "main", "trunk", "integration").
- Update the branch naming convention if your team uses a different pattern.
- If not using GitHub CLI (gh), replace the pr create command with your VCS
  platform's CLI (e.g. az repos pr create for Azure DevOps, glab mr create for GitLab).
- Adjust the secret scan regex patterns to match your project's secret naming.
- The consent gate message string can be customized but the YES/NO interaction
  pattern must be preserved — it is the safety mechanism for the orchestrator.

CHANGE 4 — MODIFIED AGENT: testing.agent.md
================================================================================

File      : .github/agents/testing.agent.md
Operation : MODIFIED (two sections updated)
What was there before: The agent wrote tests and checked coverage but had no cap
on how many iterations it would attempt, and had no defined escalation path when
80% coverage could not be reached.

CHANGE 4A — 5-ITERATION CAP ON COVERAGE LOOP
──────────────────────────────────────────────
Old behavior: "iterate until 80% coverage" — could loop indefinitely.
New behavior: Maximum 5 iterations enforced.
  - Each iteration is labeled "Coverage Iteration N of 5" in agent output.
  - After iteration 5, if coverage < 80%, the Coverage Escalation Protocol fires.
  - This prevents the agent from running indefinitely on code that structurally
    cannot reach 80% (e.g. code requiring live Oracle DB, AOP proxies, etc.).

Exact text change in Step 6 of Approach section:
  OLD: "iterate until 80% coverage"
  NEW: "Iterate toward 80% coverage (maximum 5 iterations): Parse the JaCoCo
        report, identify uncovered lines/branches, write additional tests, and
        re-run. Repeat up to 5 iterations. If 80% is not reached after 5
        iterations, trigger the Coverage Escalation Protocol — stop and report
        to the human with options."

CHANGE 4B — COVERAGE ESCALATION PROTOCOL (NEW SECTION)
────────────────────────────────────────────────────────
After 5 failed iterations, the agent STOPS and outputs a structured report:

  === COVERAGE ESCALATION REPORT ===
  Iterations completed : 5 of 5
  Current coverage     : <LINE>% line / <BRANCH>% branch
  Gap                  : <X> lines and <Y> branches still uncovered

  Uncovered Items:
    - <ClassName>.<methodName>() — Reason: <why it cannot be covered>

  Options (choose one):
    1. EXCLUDE  — Add JaCoCo exclusion in pom.xml for the class/method
    2. ACCEPT   — Accept current coverage with documented justification
    3. REFACTOR — Refactor the class for testability (specific suggestion provided)
    4. INTEGRATION TEST — Replace gap with @SpringBootTest integration test

The agent DOES NOT auto-apply any option. It waits for the human to choose.
This is intentional: coverage decisions that involve excluding code or accepting
below 80% require human judgment and must be logged in the codebase.

WHY THIS DESIGN
───────────────
Without the cap, a testing agent in an automated workflow will keep writing
tests indefinitely for code that cannot be unit-tested (Oracle-specific behavior,
Spring AOP internals, main() methods). The 5-iteration cap + 4-option escalation
gives the developer a clear decision point rather than an infinite loop.

ADAPTING TO ANOTHER PROJECT
────────────────────────────
- The 5-iteration cap is a policy number; change it to suit your team (e.g. 3 for
  fast workflows, 7 for thorough coverage).
- The 80% threshold matches the SonarQube quality gate in this project; update
  to match your project's JaCoCo minimum coverage configuration in pom.xml.
- The 4 escalation options are generic and apply to any Java project unchanged.
- Coverage exclusions list (entities, DTOs, config, Application.java) should be
  updated to match your project's exclusion policy.

CHANGE 5A — REMOVED @sql-data FROM ALL WORKFLOWS
──────────────────────────────────────────────────
@sql-data was previously referenced in some workflow descriptions as a step before
@java-api-dev. It has been removed from ALL workflow references. @sql-data still
exists as a standalone agent for schema/seed generation tasks but is no longer
wired into the automatic story or feature implementation workflows.

Reason: @sql-analysis serves the analysis role; @sql-data is an output agent
(generates DDL), not an analysis agent. Mixing them in the workflow was confusing.

CHANGE 5B — UPDATED CONSTRAINTS SECTION
────────────────────────────────────────
Two new constraints added:
  - "DO NOT skip @maven-ops — after @testing, always run @maven-ops to verify
    the build is green before @code-review."
  - "DO NOT invoke @git-ops without human consent — always present the consent
    gate summary and wait for explicit yes before committing or pushing."

CHANGE 5C — UPDATED HOW TO EXECUTE SECTION
────────────────────────────────────────────
Steps 4-6 updated to reflect the new agent chain:
  Step 4: After @testing completes, invoke @maven-ops (compile + test gate).
  Step 5: After @maven-ops passes, invoke @code-review.
  Step 6: After @code-review, present consent gate before invoking @git-ops.

CHANGE 5D — UPDATED AVAILABLE AGENTS TABLE
────────────────────────────────────────────
New rows added:
  SQL Analysis | @sql-analysis | Analyze SQL in story vs DDL; fast-pass if no SQL
  Maven Ops    | @maven-ops    | Compile, test, start/restart on port 8080
  Git Ops      | @git-ops      | Branch, commit, push, PR — requires human consent

Testing row updated:
  OLD: "Write JUnit 5 + Mockito tests"
  NEW: "Write JUnit 5 + Mockito tests; iterates up to 5 times toward 80% coverage"

CHANGE 5E — UPDATED ROUTING LOGIC TABLE
────────────────────────────────────────
New routing entries:
  "Analyze SQL / queries in story"          → @sql-analysis
  "Compile / run / check build"             → @maven-ops
  "Start / restart server on port 8080"     → @maven-ops
  "Create branch / commit / push / open PR" → @git-ops

CHANGE 5F — STORY IMPLEMENTATION WORKFLOW (7 STEPS — CORE CHANGE)
──────────────────────────────────────────────────────────────────
This is the main orchestration change. The workflow was extended from 3 steps to 7:

  OLD (3 steps):
    1. @java-api-dev
    2. @testing
    3. @code-review

  NEW (7 steps):
    1. @sql-analysis  — fast-pass if no SQL; else produce DDL mapping plan
    2. @java-api-dev  — build API using sql-analysis plan as schema reference
    3. @testing       — write tests (5-iteration coverage gate, escalate if needed)
    4. @maven-ops     — BUILD GATE: mvn compile + mvn test; STOP workflow on failure
    5. @code-review   — only runs after maven-ops green
    6. CONSENT GATE   — show branch/commit/PR summary, ask for explicit YES/NO
    7. @git-ops       — only on YES; create branch, commit, push, open PR to develop

The note "Steps 3 (@testing) and 4 (@maven-ops) are NOT optional" was added
explicitly to prevent the orchestrator from skipping them in shortened workflows.

CHANGE 5G — UPDATED BUG FIX WORKFLOW (5 STEPS)
────────────────────────────────────────────────
  OLD: @bug-fix → @testing → @code-review
  NEW: @bug-fix → @testing (5-iter) → @maven-ops → @code-review → CONSENT → @git-ops

CHANGE 5H — UPDATED "IMPLEMENT A FEATURE" WORKFLOW
────────────────────────────────────────────────────
  OLD: @java-api-dev → @testing → @code-review → @doc-gen
  NEW: @sql-analysis → @java-api-dev → @testing → @maven-ops → @code-review →
       CONSENT → @git-ops → @doc-gen

CHANGE 5I — UPDATED "DATABASE SCHEMA CHANGE" WORKFLOW
───────────────────────────────────────────────────────
  OLD: (no defined workflow)
  NEW: @sql-analysis → @api-modification → @testing (5-iter) → @maven-ops

CHANGE 5J — ADDED FAILURE HANDLING FOR @maven-ops AND @git-ops
────────────────────────────────────────────────────────────────
  @maven-ops failure: blocks @code-review and @git-ops; reports exact errors; stops workflow.
  @git-ops failure: reports exact error + provides manual PowerShell git commands
    as fallback so the developer can complete the publish manually.

ADAPTING TO ANOTHER PROJECT
────────────────────────────
- The 7-step Story Implementation Workflow is generic and works for any Spring Boot
  project. Only the agent names and tool references need to match your project's agents.
- Replace the story detection heuristics ("As a [role]", acceptance criteria, etc.)
  if your team uses a different story format (e.g. Gherkin, JIRA descriptions).
- If your project does not use GitHub CLI (gh), update the @git-ops consent gate
  output and the @git-ops agent's PR creation command.
- The CONSENT GATE pattern (display summary → ask YES/NO → only proceed on YES)
  is a human-in-the-loop safety mechanism and should be kept regardless of platform.
- The BUILD GATE pattern (@maven-ops step 4) is critical: without it, @code-review
  and @git-ops can run on broken builds. Keep it or replace with your build tool's
  equivalent (e.g. ./gradlew build for Gradle projects).
